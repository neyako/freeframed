## Context (read once)

This work spans **two repositories**:

- **FreeFrame** (this repo): `/Users/neyako/freeframed` — turbo monorepo.
  `apps/api` = FastAPI + SQLAlchemy + Postgres + MinIO/S3 + Celery/ffmpeg transcoder.
  `apps/web` = Next.js (App Router) + Tailwind, the review UI. Planned at commit `dfa0ab1`.
- **projmgmt**: `/Users/neyako/projmgmt` — single Next.js 15 / React 19 / Prisma / SQLite
  app. The production "control room": kanban pipeline, Nextcloud + NAS paths, sponsor CRM.
  Planned at commit `1905a0b`.

**Decided architecture (service bridge).** projmgmt stays the hub. FreeFrame runs as a
companion review service. The whole project tree (assets / sfx / luts / vfx / raw) lives
in **Nextcloud/NAS** (projmgmt's existing domain); only the **final/draft video** is pushed
into FreeFrame for frame-accurate review. When a project moves `Editing → Review`, projmgmt
mints a **single-video guest share** on FreeFrame and stores that URL in `Project.reviewLink`.
Reviewers never get accounts and only ever see the one sent video. The editor team uses
FreeFrame/projmgmt directly with full access.

Each plan states its **Target repo** at the top. An executor works in ONE repo per plan —
do not modify the other repo from within a plan unless the plan's Scope says so.

## Recommended sequencing

1. **001** first — self-contained mobile fix, immediate UX win, no dependencies. Ship it alone.
2. **002** next — makes FreeFrame shares reviewer-safe. Foundation the bridge relies on.
3. **003** — adds the S2S ingest endpoint that creates an asset + reviewer-safe share in one call. Needs 002's share helper.
4. **004** — projmgmt side of the bridge. Needs 002 (share shape) and 003 (endpoint) live and reachable.
5. **005** — independent projmgmt feature (tree vs final upload). Can run any time, even before 001.

## Dependency notes

- **003 requires 002** because the ingest endpoint reuses the reviewer-safe share helper
  introduced in 002 (`create_reviewer_share` in `apps/api/routers/share.py`).
- **004 requires 002 + 003** because it calls the FreeFrame ingest endpoint (003) and expects
  the reviewer-safe share semantics (002). 004 cannot be verified end-to-end until a FreeFrame
  instance with 002+003 is running and reachable from projmgmt.
- **001 and 005** have no dependencies and can be done in parallel with the rest.

## Findings considered and rejected

- *"Port FreeFrame's review engine natively into projmgmt"* — rejected by the maintainer in
  favor of the service-bridge architecture. A native port means rewriting Python→TS,
  Postgres→SQLite, MinIO→Nextcloud, and replacing the ffmpeg/HLS transcoder: months of work,
  high risk, and it throws away FreeFrame's strongest infrastructure. Not pursued.
- *"Give reviewers FreeFrame accounts with a restricted role"* — rejected in favor of guest
  share links. No reviewer onboarding, no new role to enforce repo-wide.
- *"Store the whole project tree in FreeFrame/MinIO"* — rejected; project files (sfx/luts/vfx)
  are not reviewable media and belong in Nextcloud/NAS where projmgmt already manages them.
